# 大创项目汇报：基于多智能体协同与 PoC 验证的 DApp 攻击链分析系统

> 汇报人：王涛 | 指导教师：吴嘉婧 | 中山大学软件工程学院

---

## 一、项目一句话概述

本项目面向复杂 DApp 攻击链的分析难题，提出**"多智能体协同 + PoC 重放验证"**新范式：先宏观识别攻击角色与资金流，再微观动态探索执行轨迹，最终生成 Foundry 可运行 PoC 脚本并在私链重放验证，以**可执行解释替代文本幻觉**。

---

## 二、项目背景与痛点

### 2.1 为什么做这个项目

| 痛点 | 具体表现 |
|------|----------|
| **攻击日益复杂** | 真实 DApp 攻击常由多笔交易组成（准备→攻击→获利→洗钱），跨合约、跨交易，形成完整攻击链 |
| **现有工具不足** | DAppFL 依赖静态规则剪枝，易误删关键路径；FaultSeeker 仅支持单交易分析，割裂攻击全局逻辑 |
| **大模型有幻觉** | Malo 等 LLM 方法生成的解释缺乏可执行验证，难以判断真伪 |
| **人工审计低效** | 复杂攻击需手工梳理资金流和合约调用，耗时数天到数周 |

### 2.2 我们的解决思路

> 不是对代码做通用漏洞扫描，而是针对**已发生的异常交易**，进行"由果溯因"的精准诊断。核心闭环：**假设 → 生成 → 验证**。

## 三、核心创新点（4个）

### 创新 1：多智能体协同的全链路攻击分析

将分析流程拆分为 9 个专职 Agent，各司其职：

| Agent | 职责 | 所处阶段 |
|-------|------|----------|
| TxDetailAgent | 获取交易详情、Trace、合约、事件 | 数据采集 |
| TxRoleAgent | 识别攻击交易、辅助交易、受益人 | 宏观分析 |
| Transaction Filter | 去重相似攻击模板 | 筛选 |
| TxFaultAgent | 交易级异常摘要，提供宏观线索 | 宏观分析 |
| Task Organizer | 组织当前调试计划 | 微观分析 |
| Transaction Debugger | Trace 级漏洞定位，调用工具获取证据 | 微观分析 |
| GlobalMemory Administrator | 维护跨轮次共识和事实 | 协调中枢 |
| Code Patcher | 生成 Solidity 补丁并编译 | 修复验证 |
| Transaction Judge | 判断补丁是否抵抗攻击 | 验证闭环 |

**分工的价值**：不是"堆 Agent"，而是这个任务天然有多种认知角色——角色识别、Trace 调试、补丁生成和验证需要不同上下文、不同工具权限。拆分后职责边界清楚，日志可观测，失败可回滚。

### 创新 2：语义感知的动态 Trace 剪枝（Token 管理）

**问题**：单笔交易 Trace 可达数万行函数调用，跨交易场景直接击穿大模型上下文窗口（如 30K+ tokens）。

**方案**：不是简单截断，而是基于 Trace 调用树结构做**语义感知剪枝**：
- 初始只展开有限深度（如 depth=2）
- Agent 可按需 `expand_node` 深入可疑子树
- 对低价值子树 `collapse_node` 折叠（如 precompile 地址、低 gas STATICCALL）
- 已分析节点用 `comments` 保留摘要
- 动态 Token Budget = 模型上下文窗口 - 最大输出 - safety buffer - 历史对话 - 静态 prompt

**效果**：在 SushiSwap 等复杂案例中，单案例输入从约 30K token 降到约 12K token（降低约 60%），同时保留根因定位效果。

### 创新 3：工具增强的可证据推理（非传统 RAG）

传统 RAG = `query → 向量搜索 → 检索片段 → prompt`

TracePilot 的检索 = `Trace node index / tx hash → 确定性工具调用 → 源码/状态/事件证据`

**为什么不用向量库**：
- 链上证据需要**精确定位**，不能依赖语义相似度猜测
- Trace 节点有明确 index，可以直接取源码、内部执行流、状态快照
- 审计场景要求**证据可追溯**，工具调用返回确定性数据

Agent 通过 MCP 工具按需读取：当前 Trace 树、函数源码、合约源码、内部执行流、状态快照、storage 变化、event logs、编译信息。

### 创新 4：Debug-Patch-Verify 自验证闭环

定位结果**必须**经补丁和攻击重放验证：

```
Transaction Debugger (定位漏洞函数)
        ↓
Code Patcher (生成 Solidity 补丁)
        ↓
Transaction Judge (判断补丁质量)
        ↓
  ├── VERIFIED → 输出最终报告
  ├── WRONG_ROOT_CAUSE → 回到 Debugger 重新定位
  └── INEFFECTIVE_PATCH / BROKEN_LOGIC → 修改补丁继续验证
```

Judge 区分失败类型：编译失败、初始化失败、补丁无效、根因错误、业务逻辑破坏，不同失败回写给不同 Agent。这比只输出漏洞解释更接近真实审计流程。

## 四、技术架构

### 4.1 整体架构

```
TracePilot Dashboard (React 18 + TypeScript + Ant Design + WebSocket)
        ↕  REST API + WebSocket
TracePilot Backend (FastAPI + asyncio + ThreadPoolExecutor)
        ↓
  ┌─────┴─────┐
  │  TaskManager │  (任务生命周期、队列、取消、持久化)
  └─────┬─────┘
        ↓
  Workflow Thread
  ├── DAppProcess (链上数据采集：Tenderly/RPC/Etherscan)
  └── DAppAnalyze (多 Agent Debug-Patch-Verify 闭环)
        ↓
  Persistence Layer
  ├── PostgreSQL (task_runs, task_logs)
  ├── Redis (完整日志内容缓存)
  ├── agents/logs/ (Agent 原始文件日志)
  └── data/cache/ (链上数据缓存)
```

### 4.2 技术栈一览

| 层次 | 技术 | 作用 |
|------|------|------|
| 前端 | React 18, TypeScript, Vite, Ant Design 6, lucide-react | 任务工作台、实时日志、报告渲染、导出 |
| 通信 | REST API, WebSocket | 任务管理 + 实时事件推送 |
| 后端 | FastAPI, asyncio, ThreadPoolExecutor | Web API、任务生命周期、异步协调 |
| 模型 | OpenAI-compatible API (DeepSeek/GPT/Qwen 等) | 多 Agent 推理、function calling |
| 工具 | FastMCP, 自定义 Trace 调试工具 | 源码/状态/事件按需检索 |
| 数据 | Tenderly, JSON-RPC, Etherscan API | 链上交易、Trace、合约源码 |
| 存储 | PostgreSQL, Redis, local files | 任务、日志、完整 Agent 输出 |
| 容器化 | Docker Compose | 一键启动后端 + PostgreSQL + Redis |

## 五、平台功能展示（当前已实现）

### 5.1 后端功能（✅ 已完成）

- ✅ 任务创建、取消、归档、恢复、删除（完整生命周期管理）
- ✅ 多 Agent 协同工作流（9 个 Agent，Debug-Patch-Verify 闭环）
- ✅ WebSocket 实时推送（状态、日志、心跳、背压控制）
- ✅ 三层日志持久化（Redis 缓存 + PostgreSQL 索引 + 文件日志）
- ✅ 链上数据缓存（Tenderly/RPC/Etherscan 数据本地缓存，避免重复请求）
- ✅ Docker 一键部署（docker-compose up --build）
- ✅ 149 例真实 DApp 漏洞案例数据集
- ✅ 多模型支持（DeepSeek/GPT/Qwen/Llama 等 OpenAI-compatible 模型）

### 5.2 前端功能（✅ 已完成）

- ✅ **任务管理面板**：创建、状态筛选（Pending/Running/Completed/Failed/Archived）
- ✅ **Agent State**：9 个 Agent 的事件数、工具调用数、错误数、最新摘要
- ✅ **实时日志流**：WebSocket 实时接收，支持按 Agent/Level/Message Type 过滤
- ✅ **Agent Timeline**：关键事件时间线（状态变化/工具调用/结果/错误）
- ✅ **Agent Brief**：每个 Agent 的聚合统计和最新输出摘要
- ✅ **日志详情抽屉**：点击任意日志查看完整 Markdown/JSON 内容
- ✅ **文件日志查看**：读取后端 agents/logs/ 下的 Agent 原始日志文件
- ✅ **结构化报告**：将最终报告拆分为漏洞根因、攻击路径、关键交易、补丁建议、验证结果
- ✅ **报告导出**：导出 Markdown 审计报告 + JSON evidence package
- ✅ **刷新恢复**：页面刷新后从数据库恢复任务状态和历史日志

### 5.3 数据流（端到端）

```
用户选择 DApp → POST /api/tasks
    → TaskManager 创建任务，写入 task_runs
    → 启动 Workflow Thread（线程池，不阻塞 API）
    → DAppProcess: 链上数据采集 → 资金流图 → 角色识别 → 交易筛选
    → DAppAnalyze: Debug-Patch-Verify 主循环（9 Agent 协同）
    → WebSocket 实时推送每步状态/日志
    → 任务完成，final_report 持久化到 PostgreSQL
    → 前端可导出 Markdown/JSON，可归档/删除
```

### 7.3 可能被问到的问题（需准备）

| 问题 | 回答要点 |
|------|----------|
| 和 DAppFL/Malo 的区别？ | DAppFL 是单交易静态分析，我们是跨交易动态分析 + PoC 验证；Malo 只生成文本解释，我们生成可执行 PoC |
| 为什么用 9 个 Agent 而不是一个大 Prompt？ | 不同阶段需要不同上下文和工具权限，拆开职责清晰、日志可观测、失败可回滚 |
| 剪枝会不会丢失关键证据？ | 不是简单截断，是基于调用树结构的语义感知折叠 + 按需展开；保留了工具展开和回滚机制 |
| 你的核心贡献是什么？ | Token 剪枝策略设计 + MCP 多智能体协同调用 + 工程化平台搭建（WebSocket/日志/归档/导出） |
| 大模型幻觉问题怎么解决？ | 通过工具调用获取确定性链上证据，通过 PoC 重放验证结论，通过 Judge 区分不同失败类型 |
| 平台能替代人工审计吗？ | 当前定位是辅助工具，降低人工审计成本，并通过补丁验证提高结论可靠性 |

## 八、项目亮点总结（汇报时重点强调）

- 提出"多智能体协同 + PoC 重放验证"新范式，解决跨交易攻击链分析难题
