# TracePilot 前后端工程汇报简稿

老师、师兄好，我这阶段主要做的是 TracePilot 的前后端工程化落地。原来的系统更偏科研脚本：Agent 在后台跑，输出主要依赖终端和本地日志，不方便实时观察、复盘和演示。我现在做的工作，是把一次 DApp 漏洞分析封装成一个可交互的 Web 任务。

后端部分，我用 FastAPI 封装了任务创建、任务状态查询、取消、归档、删除、日志查询和 WebSocket 实时推送等接口。任务执行是长耗时流程，所以没有直接阻塞 FastAPI 主事件循环，而是交给 ThreadPoolExecutor 后台执行；后台 Agent 产生的日志再通过线程安全回调写回主事件循环，由 WebSocket 推送给前端。

日志这块我做了结构化分层。第一层是任务控制事件，比如 pending、running、completed、failed；第二层是 Agent 运行日志，比如每个 Agent 的阶段、输入摘要、输出摘要；第三层是工具调用日志，比如获取 Trace、源码、状态快照和事件日志；第四层是最终结果和补丁报告。每条日志还按 `info/debug/warning/error` 区分级别，按 `text/markdown/tool/result` 区分展示类型。这样前端可以按 Agent、级别和类型过滤，而不是把所有文本混在一起。

持久化方面，我用了 PostgreSQL、Redis 和本地文件三层。PostgreSQL 保存任务元数据、日志索引和最终报告；Redis 用 `log_id` 缓存完整日志内容，前端点击详情时再按需拉取；本地文件保留每个 Agent 的原始日志，方便事后复盘。这样既能实时看，也能刷新恢复，还能导出报告。

前端部分，我用 React + TypeScript + Ant Design 做了 Dashboard。首页负责创建任务、查看历史记录、取消或归档任务；详情页展示 9 个 Agent 的状态、实时日志流、时间线、文件日志和结构化报告。最终报告会拆成漏洞根因、攻击路径、关键交易、补丁建议、验证结果几个区块，并支持导出 Markdown 和 JSON evidence package。

部署上，我引入了 Docker Compose，把 FastAPI 后端、PostgreSQL 和 Redis 放在一套编排里。这样新同学只需要配置 `.env`，然后 `docker-compose up --build` 就能启动后端依赖环境，避免因为 Python 版本、数据库或 Redis 配置不同导致跑不起来。

等下我会用 SushiSwap 漏洞做演示。这个案例属于跨交易漏洞分析场景，Trace 和链上背景都比较典型。现在前端在选择 SushiSwap 后，可以点 `Context` 按钮查看漏洞背景、根因函数、攻击交易哈希和外部参考链接。这样在 Agent Agent 的定位过程。

目前平台已经支持从任务创建、实时观测、日志详情、历史恢复到报告导出的一整套流程。后续我准备继续补三个方向：第一，日志流做虚拟滚动和分页恢复，支持更长任务；第二，把 DApp 列表和漏洞背景从前端 JSON 迁移到后端 `/api/dapps` 接口；第三，创建任务时支持选择模型、剪枝深度、最大轮次等参数，让系统更像一个可配置的实验和审计平台。
