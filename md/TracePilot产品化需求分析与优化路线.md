# TracePilot 产品化需求分析与优化路线

> 目标：将 TracePilot 从科研原型和内部演示工具，逐步演进为面向链上攻击复盘、漏洞根因定位和补丁验证的工程化产品。

---

## 1. 产品定位

TracePilot 不应直接定位为“又一个 AI 智能合约审计器”。这类产品通常面向未上线或待审计合约，主要回答：

> 这个合约可能有哪些漏洞？

TracePilot 更适合切入一个更窄但更有差异化的方向：

> 给定真实链上攻击交易，自动复盘攻击路径、定位根因函数、生成证据链，并给出补丁建议和验证结果。

因此，推荐产品定位为：

> **TracePilot：面向真实链上攻击的 AI 取证与跨交易漏洞根因分析平台。**

核心价值不是简单输出一份“漏洞列表”，而是把复杂链上攻击变成：

- 可理解的攻击时间线
- 可追溯的证据链
- 可定位的根因函数
- 可验证的补丁建议
- 可导出的复盘报告

---

## 2. 当前基础

目前项目已经具备产品化雏形：

- 后端已支持 FastAPI 任务服务化，包括任务创建、查看、取消、归档、删除。
- WebSocket 已支持多 Agent 分析过程的实时推送。
- 日志已经结构化为 `agent / level / message_type / log_id`。
- PostgreSQL 保存任务元数据、日志索引和最终报告。
- Redis 缓存完整日志内容，支持点击详情按需加载。
- 前端 Dashboard 已支持任务列表、Agent 状态、日志流、Timeline、文件日志、结构化报告和导出。
- Docker Compose 已支持后端、PostgreSQL、Redis 一键启动。
- 已有 149 例真实 DApp 漏洞数据集，并有 ISSTA 2026 论文背书。

这些基础说明 TracePilot 已经不只是算法原型，而是具备“可交互、可观察、可复盘”的平台基础。

## 3. 与通用 AI 审计工具的差异

| 维度 | 通用 AI 审计工具 | TracePilot 推荐定位 |
|------|------------------|---------------------|
| 输入 | 合约源码、仓库、合约地址 | 真实攻击交易、链名、合约地址、可选源码 |
| 核心问题 | 合约可能有什么漏洞 | 这次攻击为什么成功 |
| 分析依据 | 源码、静态规则、LLM 推理 | 交易 Trace、状态变化、事件日志、源码、LLM 推理 |
| 输出 | 漏洞列表、严重性、修复建议 | 攻击路径、根因函数、证据链、补丁验证 |
| 适用场景 | 上线前扫描、CI 预审 | 攻击复盘、安全教学、事故响应、审计训练 |
| 竞争关键 | 漏洞覆盖率、误报率 | 真实攻击解释能力、证据可追溯性、跨交易分析能力 |

TracePilot 应避免和成熟静态分析工具正面竞争，而应突出“链上证据驱动的漏洞定位”。

## 4. 面向角色的产品需求

### 4.1 安全研究员 / 审计员

核心诉求：

- 快速理解真实攻击如何发生。
- 找到根因函数和关键调用路径。
- 验证模型结论是否有链上证据支撑。
- 导出可复盘的报告。

需要的功能：

- 攻击交易输入与自动 Trace 拉取。
- 攻击路径时间线。
- 关键函数和 Trace 节点定位。
- Storage Change / Event Log / Function Call 证据聚合。
- 最终报告导出。

### 4.2 智能合约开发者

核心诉求：

- 看懂为什么自己的合约被攻击。
- 知道应该改哪一个函数。
- 知道补丁是否真的能挡住攻击。

需要的功能：

- 根因函数定位。
- 修复建议。
- 补丁验证结果。
- 关键代码片段和风险解释。
- 不需要默认展示大量 Agent 原始日志。

### 4.3 学生 / 新手审计学习者

核心诉求：

- 通过真实案例学习攻击模式。
- 理解每笔交易在攻击链中的作用。
- 学习如何从 Trace 推导漏洞根因。

需要的功能：

- Learn Mode：逐步解释攻击阶段。
- 背景信息展示，包括漏洞报告链接、根因、攻击交易哈希。
- 可视化攻击时间线。
- 术语解释和关键节点说明。

### 4.4 项目负责人 / 老师 / 汇报对象

核心诉求：

- 快速看到系统价值。
- 不陷入过多日志细节。
- 看到可演示、可复现、可导出的结果。

需要的功能：

- 一页式 Summary。
- 关键指标：分析状态、发现根因、验证结果、证据数量。
- 可导出报告。
- Demo Case，例如 SushiSwap。

## 5. 当前不足

### 5.1 信息太多，但信息层级不够清晰

当前系统会产生大量 Agent 输出、工具调用结果和中间日志。对开发调试有价值，但对最终用户来说容易造成负担。

需要调整为：

> 最终报告是主产品，Agent 日志是证据层和调试层。

默认展示结论和证据，原始日志折叠到高级视图。

### 5.2 报告和证据没有强绑定

目前最终报告能说明漏洞，但报告中的关键结论还不能直接跳转到对应 Trace 节点、交易、事件日志或源码位置。

产品化后应做到：

- 每个根因结论都有证据。
- 每条证据可以点击查看来源。
- 用户能判断“AI 为什么这么说”。

### 5.3 输入方式仍偏数据集驱动

目前更适合跑已有 DApp JSON。产品化后需要支持更自然的输入：

- 链名
- 攻击交易哈希
- 合约地址
- 可选攻击者地址
- 可选源码仓库或源码文件

### 5.4 缺少置信度和失败说明

安全产品不能只输出“结论”，还应说明：

- 证据是否充分
- 哪些交易未成功分析
- 哪些 Agent 结果存在分歧
- 补丁验证是否成功
- 未验证部分是什么

### 5.5 任意项目审计能力还不足

如果用户只上传一个 GitHub 仓库，没有真实攻击交易，TracePilot 当前优势无法完全发挥。

后续若要支持“任意项目”，建议接入：

- Slither / Aderyn 静态分析
- Foundry / Hardhat 编译与测试
- Foundry fork replay
- Echidna / Foundry fuzzing

这部分应作为第二阶段能力，而不是第一阶段 MVP。

## 6. 产品优化方向

### 6.1 报告优先，而不是日志优先

当前 Dashboard 可以保留日志能力，但默认入口应调整为：

1. 结论摘要
2. 攻击路径
3. 根因函数
4. 证据链
5. 补丁建议
6. 验证结果
7. Agent 原始日志

这样不同用户都能快速进入适合自己的层级。

### 6.2 Evidence Graph：证据链视图

将最终报告中的核心结论挂接到证据：

```text
根因：bridgeFor 默认使用 WETH 作为中间路由
证据：
- Tx 1 创建低流动性 DIGG-WETH 池
- Tx 2 调用 convert 触发费用兑换
- Trace Node #128 调用 bridgeFor()
- Event Swap 显示大量 DIGG 被倾倒
- Storage Change 显示池子储备比例异常
```

这个能力是 TracePilot 最重要的产品差异化之一。

### 6.3 Attack Replay Timeline：攻击复盘时间线

将原始日志和 Trace 转化为攻击语义阶段：

```text
准备阶段：创建低流动性池
操纵阶段：触发 fee conversion
价格扭曲：池子储备比例异常
获利阶段：低价换回资产
根因定位：bridgeFor 缺少路由约束
修复验证：补丁后攻击无法复现
```

它比普通日志流更适合展示，也更适合教学和汇报。

### 6.4 多模式视图

建议设计三种视图：

| 模式 | 面向用户 | 默认展示 |
|------|----------|----------|
| Report Mode | 老师、项目负责人、普通开发者 | 摘要、根因、修复、验证 |
| Learn Mode | 学生、新手审计员 | 逐步解释攻击过程 |
| Auditor Mode | 安全研究员、审计员 | 证据链、Trace 节点、源码、日志 |

Raw Agent Logs 仍然保留，但不作为默认主视图。

### 6.5 置信度和覆盖度

为每次分析增加质量指标：

- 根因置信度
- 证据充分性
- 已分析交易数 / 总交易数
- 补丁验证状态
- Agent 失败数
- 未覆盖风险

这样即使模型不完美，也能让用户知道结果边界。

## 7. 分阶段建设路线

不建议一开始把所有功能都做完。推荐分三阶段推进。

### 阶段一：Demo MVP，面向组会和实验室内部试用

目标：

> 让 SushiSwap / Audius 等典型案例跑起来，并把结果讲清楚。

优先功能：

- 支持选择已有 DApp 案例运行。
- 支持查看漏洞背景 Context。
- 优化最终报告展示。
- 将最终报告拆成固定区块：根因、攻击路径、关键交易、补丁建议、验证结果。
- Agent 日志默认折叠，保留详情抽屉。
- 支持 Markdown / JSON 报告导出。

暂不做：

- 任意 GitHub 仓库审计。
- 完整登录系统。
- 复杂权限管理。
- 在线收费。

阶段一成功标准：

- 能稳定演示 2-3 个经典攻击案例。
- 老师和新同学能在 5 分钟内看懂一次分析结果。
- 报告能导出并用于复盘。

### 阶段二：攻击交易输入版

目标：

> 从数据集驱动走向真实用户输入。

优先功能：

- 输入链名 + 攻击交易哈希。
- 自动拉取交易 Trace 和基础元信息。
- 自动识别相关交易、攻击者、受害合约。
- 生成攻击时间线。
- 生成可点击证据链。
- 对失败任务提供明确失败原因。

阶段二成功标准：

- 用户不需要手工准备 JSON。
- 能分析部分真实链上攻击交易。
- 报告结论可以追溯到链上证据。

### 阶段三：项目预审与 CI 扩展

目标：

> 扩展到未发生攻击的普通项目安全预审。

优先功能：

- 支持 GitHub / Foundry / Hardhat 项目输入。
- 接入 Slither / Aderyn 静态分析。
- 接入 Foundry fork test 或 fuzzing。
- 将静态发现与 TracePilot 的补丁验证流程结合。
- 支持 CI 报告和阈值阻断。

阶段三成功标准：

- 不仅能复盘攻击，也能对普通项目做初步安全预审。
- 能输出机器可读报告，便于接入研发流程。

短期最值得做的功能：

| 功能 | 优先级 | 说明 |
|------|--------|------|
| 漏洞背景 Context 按钮 | P0 | 已具备基础，适合演示 |

| 交易哈希输入 | P2 | 从案例演示走向真实输入 |

## 9. 推荐演示话术

可以这样介绍 TracePilot：

> 传统 AI 审计工具更多回答“这个合约可能有什么漏洞”。TracePilot 关注的是另一个问题：真实攻击已经发生后，攻击路径是什么、根因函数在哪里、证据是否充分、补丁能否阻断攻击。我们把多 Agent 分析过程组织成可复盘的攻击时间线和证据链，而不是只输出一堆日志。

面向 SushiSwap 演示时可以这样讲：

> 这里我选择 SushiSwap 案例。系统先展示漏洞背景，包括攻击时间、交易哈希、根因函数和公开报告链接。任务运行后，多个 Agent 会从交易 Trace、函数调用、状态变化和事件日志中逐步定位攻击路径。最终报告会归纳为根因、攻击路径、关键交易、补丁建议和验证结果。底层 Agent 日志仍然可以展开，用于解释系统为什么得出这个结论。

---

## 10. 总结

TracePilot 的产品化方向不应是“替代所有审计工具”，而应先聚焦：

> 真实链上攻击复盘、跨交易漏洞定位、证据链解释和补丁验证。

短期目标不是一次性完成商业产品，而是先做出一个稳定、漂亮、可演示的 MVP：

- 用户能选择或输入一个攻击案例。
- 系统能跑完整分析流程。
- 结果以结构化报告呈现。
- 关键结论能追溯到证据。
- Agent 日志作为专家模式保留。

这样既能保留论文工作的技术深度，也能形成普通 AI 审计工具难以替代的产品竞争力。
