# TracePilot 前后端工作汇报

> 汇报人：王涛 | 2026.05.13

## 零、汇报主线

本阶段工作的核心目标，是把 TracePilot 从“科研脚本式输出”推进到一个**可交互、可观察、可复盘、可部署**的工程化平台。围绕这个目标，我主要做了三类工作：

1. **任务服务化**：用 FastAPI 封装任务创建、执行、取消、归档、删除、报告查询等能力，让原本命令行触发的分析流程变成可被前端调用的 Web 服务。
2. **过程可观察**：把多 Agent 的分析过程拆成结构化日志事件，通过 WebSocket 实时推送到前端，并用 PostgreSQL + Redis + 本地文件做持久化，支持刷新恢复和后续复盘。
3. **部署可复现**：引入 Docker Compose，把后端、PostgreSQL、Redis 编排到同一套运行环境中，降低新同学本地启动和联调成本。

一句话来说：这部分工作不是只做一个“日志页面”，而是给多 Agent 漏洞定位系统补上了任务生命周期管理、实时观测、日志追溯和容器化部署这几块工程能力。

### 1.2 已实现的接口

| 方法 | 路径 | 功能 |
|------|------|------|

|------|------|------|
| `WS` | `/ws/{task_id}` | 实时推送任务状态 + Agent 日志流 |

### 1.3 TaskManager 核心设计

位于 [task_manager.py](app/task_manager.py)，是整个后端的调度中枢：

- **任务创建**：生成 UUID，创建 `TaskResponse` 和事件队列，持久化到 PostgreSQL
- **任务启动**：将工作流放入 `ThreadPoolExecutor`（最大 3 个并发），避免长任务阻塞 FastAPI 主事件循环
- **线程桥接**：工作流在子线程执行，通过 `loop.call_soon_threadsafe` 安全回写主事件循环
- **取消机制**：`threading.Event` 信号 + `future.cancel()` 双保险，工作流内部轮询 `cancellation_checker`
- **归档/恢复**：只改 `archived` 字段，不删除数据；首页默认查询过滤归档任务
- **删除**：级联删除 task_runs 记录 + task_logs 记录 + Redis 缓存日志
- **启动恢复**：`_load_persisted_tasks()` 从 PostgreSQL 恢复历史任务到内存，中断的 PENDING/RUNNING 任务自动标记为 FAILED

### 1.4 WebSocket 通信协议

位于 [main.py](app/main.py) 的 `websocket_endpoint` 函数：

**服务端 → 客户端消息类型：**

| type | 用途 |
|------|------|
| `CONNECTED` | 连接建立确认 |
| `TASK_STATUS` | 任务状态同步（pending/running/completed/failed/error/archived） |
| `LOG` | Agent 日志事件（含 agent、level、message_type、log_id） |
| `TASK_FINAL` | 任务最终完成通知 |
| `PING` / `PONG` | 心跳探测（20s 间隔 + 120s 超时） |
| `HEARTBEAT_TIMEOUT` | 心跳超时告警 |
| `LOG_DROPPED` | 背压丢弃通知（高吞吐时丢弃普通日志，保留控制事件） |

**客户端 → 服务端：**

| type | 用途 |
|------|------|
| `PING` | 主动心跳 |
| `PONG` | 心跳响应 |

**工程细节：**
- 解耦发送与消费：`pump_task_events` 从 TaskManager 队列消费 → 写入 `outbound_queue` → `sender` 协程异步发送
- 背压控制：outbound_queue 满时优先丢弃 LOG 事件，保留 TASK_STATUS / TASK_FINAL 等控制事件
- 同步重连：前端重连时，旧连接被主动关闭（code 1000），保证同 task_id 只有一个连接
- 已完成任务回放：连接建立时若任务已结束，自动从 PostgreSQL 加载历史日志推送给前端

### 1.5 数据持久化

| 存储 | 表/键 | 内容 |
|------|-------|------|
| PostgreSQL | `task_runs` | task_id, dapp_name, status, final_report, error, archived, created_at |
| PostgreSQL | `task_logs` | task_id, log_id, agent, level, message_type, message, full_content, timestamp |
| Redis | `log:{log_id}` | 完整日志内容（TTL 3600s） |

启动时自动执行 schema patch（`init_db()`），补齐新增字段，兼容已有数据库。

### 1.6 日志分层与分类设计

为了让前端既能“实时看进度”，又能“事后查证据”，我把日志按三个维度拆分：

#### 按事件来源分层

| 层级 | 来源 | 典型内容 | 前端展示方式 |
|------|------|----------|--------------|
| 任务控制层 | `TaskManager` | `TASK_STATUS`、`TASK_FINAL`、取消、归档、失败原因 | 顶部状态、任务列表状态、连接状态 |
| Agent 运行层 | `AgentLogger` | 每个 Agent 的输入摘要、输出摘要、推理阶段、错误告警 | Agent State、LogStream、Timeline |
| 工具调用层 | Tool / MCP / Tenderly | 工具调用名称、参数摘要、Trace 节点详情、状态快照、事件日志 | `tool` 类型日志、详情抽屉 |
| 结果报告层 | Fault / Patch / Final Report | 漏洞根因、攻击路径、补丁建议、验证结果 | 结构化报告、导出报告 |

#### 按日志级别分类

| level | 用途 |
|-------|------|
| `info` | 正常进度、模型输出、阶段切换 |
| `debug` | 中间上下文、输入摘要、调试信息 |
| `warning` | 可恢复异常、格式解析失败、上下文截断提醒 |
| `error` | 任务失败、API 异常、工具调用异常 |

#### 按消息类型分类

| message_type | 用途 | 为什么要拆 |
|--------------|------|------------|
| `text` | 普通短文本 | 适合列表快速扫描 |
| `markdown` | 格式化长文本 | 适合模型输出、阶段说明、报告片段 |
| `tool` | 工具调用事件 | 单独统计 Agent 调用了哪些工具 |
| `result` | 阶段结果/最终结果 | 便于前端提取关键节点和生成 Timeline |

每条日志都有 `log_id`。短摘要进入 WebSocket 和 PostgreSQL，完整内容同时写入 Redis，前端点击详情时再按 `log_id` 拉取完整内容，避免主日志流被大段 Markdown 或 JSON 淹没。

### 1.7 日志推送链路

实时日志从后端到前端的完整链路如下：

```
AgentLogger._send_to_callback()
        ↓ 生成 LogMessage(agent / level / message_type / log_id)
Redis.set_log(log_id, full_content) + PostgreSQL.task_logs 持久化
        ↓
TaskManager.thread_safe_log_callback()
        ↓ loop.call_soon_threadsafe 写入 task_queue
websocket_endpoint.pump_task_events()
        ↓ normalize_queue_message()
outbound_queue（带背压控制）
        ↓
sender 协程 send_json()
        ↓
前端 WebSocketService 归一化为 TaskEvent
        ↓
LogStream / Agent State / Timeline / StructuredReport 渲染
```

这里的关键点是：Agent 分析流程运行在线程池里，而 WebSocket 发送运行在 FastAPI 的 asyncio 事件循环里，所以我用 `loop.call_soon_threadsafe` 做线程安全桥接，避免子线程直接操作异步队列导致并发问题。

### 1.8 任务生命周期

```
POST /api/tasks (Create)
        ↓
   [PENDING]
        ↓ start_task → ThreadPoolExecutor
   [RUNNING]
        ↓ 工作流完成/异常
   [COMPLETED] or [FAILED]
        ↓
   ├── Archive (POST /archive) → 隐藏但保留
   │     └── Unarchive (POST /unarchive) → 恢复
   └── Delete (DELETE) → 物理删除记录+日志+Redis
```

- **Cancel**：可在 PENDING/RUNNING 状态调用，设置取消信号 + 取消 Future
- **超时清理**：[task_cleanup.py](app/task_cleanup.py) 每 5 分钟扫描一次，超过 2 小时的 RUNNING 任务自动标记为 FAILED

### 2.2 已实现功能清单

#### 任务管理（TaskList 页面）
- 创建任务：下拉选择 DApp（自动扫描 `src/data/*.json` 生成列表）
- 任务列表表格：DApp 名、状态标签、耗时、创建/完成时间、操作按钮
- 三重筛选：
  - 状态筛选：All / Pending / Running / Completed / Failed
  - 视图筛选：Active / Archived / All
- 指标卡片：任务总数、运行中数、已完成数、失败数
- 5 秒自动轮询刷新任务列表
- 批量操作：单次可多选 DApp 创建任务

#### 任务工作台（Dashboard 页面）
- **Agent State**（左侧面板）：展示 9 个 Agent 的事件数、工具调用数、结果输出数、错误/告警数、最新输出摘要；点击可联动过滤日志流
- **实时日志流**：WebSocket 实时接收，支持按 Agent、Level（info/warning/error/debug）、Message Type（text/markdown/tool/result）过滤；默认自动滚底，可暂停
- **Agent Timeline**：从事件流提取关键节点（状态变化、工具调用、结果、错误），按时序展示；过滤低价值 debug 噪声
- **Agent Brief**：每个 Agent 的聚合统计（事件数、工具调用数、结果输出数、最新摘要）
- **日志详情抽屉**（LogDetailDrawer）：点击日志查看完整 Markdown/JSON 内容，避免主页面被大段文本淹没
- **文件日志查看**（AgentFileLogs）：读取后端 `agents/logs/` 下的原始 Agent 日志文件，支持按行滚动
- **结构化报告**（StructuredReport）：
  - 自动解析 Markdown 报告，拆分为 5 个区块：漏洞根因、攻击路径、关键交易、补丁建议、验证结果
  - 标题/关键词 fallback 提取
  - Original Raw Report 保留完整上下文
- **报告导出**：
  - 导出 Markdown 审计报告（`.md`）
  - 导出 JSON evidence package（`.json`），含任务元数据、报告区块、Agent 摘要、事件流
  - 纯前端实现（Blob + URL.createObjectURL），不依赖后端接口
- **连接状态指示器**：WebSocket 连接状态实时显示（绿色/红色图标）
- **刷新恢复**：页面刷新后先拉 REST API 获取任务状态，再重连 WebSocket；已结束任务自动加载历史日志

#### WebSocketService 前端设计（[WebSocketService.ts](src/services/WebSocketService.ts)）

- **单例模式**：全局唯一实例，避免重复连接
- **自动重连**：断线后指数退避重连（1.5^n × 2s，最大 30s，最多 10 次）
- **终止判断**：收到 `TASK_FINAL` 或终端状态（completed/failed）后停止重连
- **事件归一化**：将不同消息类型统一为 `TaskEvent` 类型
- **事件历史**：缓存最近 5000 条事件，支持新组件挂载时回放
- **心跳响应**：收到服务端 PING 自动回复 PONG

### 2.3 前端不足

| 问题 | 影响 | 原因 |
|------|------|------|
| 超长任务事件累积 5000 条上限，旧事件会被丢弃 | 看不到任务早期的完整日志 | 前端内存限制，没有做虚拟滚动或分页 |
| Dashboard 刷新后，实时日志流只能看到连接后的新事件 + 数据库恢复的旧事件，中间有断连空白 | 如果任务运行中断开 WebSocket 超过缓冲区窗口，部分日志永久丢失 | 缺少按时间范围从数据库恢复日志的机制 |
| 日志列表没有虚拟滚动 | 超长日志流（几千条）时滚动可能卡顿 | 目前用简单的 slice 截断（5000 条） |
| DApp 列表从 `src/data/*.json` 扫描，硬编码在前端 | 新增 DApp 需要修改前端代码重新部署 | 后端尚缺 `/api/dapps` 接口 |
| `Archived` 筛选在前端做二次过滤 | 数据量大时前端需拉全量再过滤 | 后端 `include_archived` 参数粒度不够细 |
| 创建任务暂时没有参数配置（如选择模型、设置剪枝深度等） | 无法在 UI 上控制分析策略 | TaskCreateRequest 暂只有 dapp_name |
| Ant Design 6 与部分 lucide-react 图标风格不完全统一 | 视觉一致性有提升空间 | 渐进迁移中 |

### 3.3 Docker 容器化设计

后端通过 Docker Compose 一键启动三个核心服务：

| 服务 | 镜像/来源 | 作用 |
|------|-----------|------|
| `backend` | 本项目 `Dockerfile` | FastAPI 后端、任务调度、多 Agent 工作流 |
| `postgres` | `postgres:15-alpine` | 持久化任务元数据、日志索引、最终报告 |
| `redis` | `redis:7-alpine` | 缓存完整日志内容，支持按 `log_id` 快速读取 |

引入 Docker 的主要原因：

- **环境一致**：Python 版本、系统依赖、数据库、Redis 都固定，减少“我电脑能跑、别人跑不通”的问题。
- **一键联调**：老师或师兄只需要 `.env` + `docker-compose up --build`，即可启动后端依赖环境。
- **服务编排**：`depends_on` + PostgreSQL healthcheck 保证数据库 ready 后再启动后端。
- **数据保留**：PostgreSQL 和 Redis 使用 volume，即使容器重启，任务记录和缓存数据也不会立刻丢失。
- **开发友好**：代码目录通过 volume 挂载进容器，便于本地改代码后快速验证。
- **安全配置**：LLM Key、数据库地址、Redis 地址迁移到 `.env`，避免敏感配置硬编码进代码仓库。

- 后端：`.env`（`DATABASE_URL` / `REDIS_URL` / `LLM_NAME` / `LLM_API_KEY` 等）

## 五、总结

### 后端（`app/`）
- 基于 FastAPI 构建了完整的任务管理 Web 服务层
- 实现了 10 个 REST 端点 + 1 个 WebSocket 端点
- TaskManager 提供任务全生命周期管理（创建/取消/归档/恢复/删除）
- WebSocket 有心跳、背压控制、断连恢复、历史日志回放能力
- 数据三层存储：PostgreSQL（元数据+索引）+ Redis（大段日志缓存）+ 本地文件（Agent 原始轨迹）

### 前端（Dashboard）
- 基于 React 18 + TypeScript + Ant Design 6 + Vite 构建
- 2 个页面、9 个组件、2 个服务层模块
- 实现任务管理、实时日志流、Agent 状态面板、时间线、文件日志、结构化报告、报告导出
- WebSocket 单例服务支持自动重连、事件历史缓存、归一化

### 一句话概括

> 把多 Agent 分析过程的"脚本式输出"变成了**可交互、可观察、可复盘、可导出**的工程化平台。
